## Context
Currently, blocking is implicit — computed from dependency status at TODO generation time. There is no way to mark a task as blocked for non-dependency reasons such as waiting on external input, a design decision, or an upstream team. Users must either add a note manually or create a fake dependency.

Adding an explicit `blocked` status (or a `blocked_reason` field) would let humans express real-world blocking without abusing the dependency graph.

## Acceptance Criteria
- [ ] `status: blocked` is a valid status value in validation
- [ ] `blocked_reason` field is required when status is `blocked` (like `cancelled_reason` for `cancelled`)
- [ ] `blocked` tasks appear in TODO.md with their reason displayed
- [ ] `blocked` tasks are excluded from "Immediate starts" in TODO.md
- [ ] Status ranking places `blocked` appropriately (after `todo`, before `done`)
- [ ] `track-common.sh` updated: status rank, terminal status check excludes blocked
- [ ] `track-validate.sh` updated: accept `blocked`, require `blocked_reason`
- [ ] `track-todo.sh` updated: render blocked tasks distinctly
- [ ] `skills/work/SKILL.md` updated: document blocked status in the lifecycle
- [ ] All script changes mirrored in scaffold copies
- [ ] Tests added for blocked status validation
- [ ] Existing tests still pass

## Notes
Design decision: `blocked` as a status value (not a separate field) keeps the state machine simple — a task is either todo, active, review, blocked, done, or cancelled. The `blocked_reason` field mirrors the `cancelled_reason` pattern.

Consider whether blocked tasks should still be linkable to PRs. Probably not — if there's a PR, the task should be active/review.
