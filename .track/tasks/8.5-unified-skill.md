## Context

Original plan was to merge work/create/decompose into one 7-mode monolith skill. This is the wrong architecture. The skill framework already handles intent-based routing via descriptions. A full merge creates context window bloat, mode drift risk, and lost separation of concerns.

Revised approach — three targeted changes:

### 8.5a: Add link and context modes to `skills/work/SKILL.md`

These are sub-operations that naturally occur during work sessions:
- **link** — emit `track.link` event to JSONL log for retroactive attribution
- **context** — append to task's `## Notes` section

Add to work's Operating Modes, Definition of Done, Closing Message Matrix, and Do Not sections.

### 8.5b: (Optional) Add thin `skills/track/SKILL.md` dispatcher

- Auto-discovers in `.track/` repos
- Parses user intent, delegates to work/create/decompose
- ~30 lines, no duplicated logic
- Gives the "single entry point" UX without merging content

### 8.5c: Drop status mode

"What's the status?" = run `track-todo.sh`. Not a skill-level operation. Already covered by work's empty mode and the todo skill.

### Why not merge

| Concern | Monolith risk | Targeted approach |
|---------|--------------|-------------------|
| Context window | 7 modes = massive SKILL.md | Each skill stays focused |
| Mode drift | LLM regresses over long output | Smaller skills = less drift |
| Separation of concerns | Create and decompose are distinct workflows | Keep them as standalone skills |
| Intent routing | Duplicates what skill framework does | Framework handles it natively |

Hook-aware lifecycle: when git hooks are detected, skip manual status writes for events the hooks already emit.
