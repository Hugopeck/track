## Context

The `skills/work/SKILL.md` skill is Track's core operating protocol — it teaches agents the full task lifecycle from reading state through implementation to PR handoff. At ~400 lines, it is the most complex skill in the project and the most frequent source of agent failures.

### Current structure

The skill has three operating modes (`pick`, `resume`, `empty`), two quick operations (`link`, `context`), a provisional PR lifecycle with explicit script calls, a closing message matrix with four templates, and a detailed "Do Not" section. It covers: mode classification, task selection, overlap detection, branch/worktree setup, `track-start.sh` and `track-ready.sh` invocation, draft PR creation, acceptance criteria verification with file+line citations, plan persistence, and status management.

### Observed failure patterns

- Agents skip `track-start.sh` or `track-ready.sh` calls, causing status drift that downstream validation catches
- Agents misclassify modes (pick vs resume), especially when discovering in-progress work
- Agents open ready PRs instead of draft PRs, skipping the `active` state entirely
- Agents fabricate file+line citations for acceptance criteria verification
- Agents reproduce closing message templates poorly or waste tokens pattern-matching them
- `link` and `context` quick operations add branching logic at the skill entry point, increasing classification errors
- The `pick` mode has agents autonomously choosing tasks, which is not a desired feature — users or orchestrators (e.g., Conductor) assign work

### Architecture observation: agent vs automation boundary

The current design places lifecycle management responsibility on the agent via skill instructions. The agent must call the right scripts at the right moments. When it doesn't, the automation layer (validate, pr-lint, status-sync) catches the inconsistency and blocks the PR.

An alternative boundary exists: `track-status-sync.yml` already infers `review` from the `ready_for_review` PR event without needing the agent to call `track-ready.sh`. Similarly, `active` could potentially be inferred from a first push to a task branch. This would shift status management entirely to hooks and workflows, reducing the skill's scope to: read task context, work in the right branch, open a PR with `Track-Task: {id}`, and write notes.

### Current automation that already handles status

- `track-status-sync.yml` fires on `opened`, `reopened`, `ready_for_review`, `converted_to_draft`, `closed` — maps each to a task status
- `track-complete.yml` fires on merged PR close — writes `done`, cascades dependency unblocks
- `post-commit` hook writes events to JSONL log
- `commit-msg` hook enforces conventional commits
- `track-reconcile.sh` repairs safe status drift after the fact

### What the skill currently teaches that agents do need

- How to read a task file (frontmatter fields, `## Context`, `## Notes`)
- What branch to use and how worktrees work
- That `Track-Task: {id}` must be in the PR body for linkage
- To append discoveries to `## Notes` during work, not after
- That `files:` scopes exist and overlapping active tasks cause merge conflicts

### Related tasks

- Task 1.5 (move support scripts out of skills/) affects the file layout referenced by the skill
- Task 1.6 (pre-push validation hook) could catch invalid state before push, reducing the need for agent-side validation calls

### Scope of this planning task

Produce a design doc that maps out what the redesigned skill looks like, what moves to automation, and how existing repos using the current skill would migrate. This task is planning only — no code changes.
