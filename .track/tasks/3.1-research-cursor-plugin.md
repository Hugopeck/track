## Context
Cursor's current project-level instruction system centers on project rules in `.cursor/rules`, with legacy `.cursorrules` support still available but deprecated. Track needs a concrete gap analysis and port plan before implementation work starts.

## Research questions
1. What exact file structure and rule types should Track use under `.cursor/rules`?
2. Which existing Track guidance belongs in always-applied rules vs narrower task-specific rules?
3. What parts of the current Claude Code skill workflow map cleanly to Cursor rules, and what does not?
4. What local testing path should we use before any marketplace or distribution step?
5. What is the smallest viable Cursor integration for launch?

2026-03-28 — Researched Cursor's current `.cursor/rules` conventions and wrote the private plan at `.track/plans/3.1-cursor-rules-research.md`.

Key conclusions:
- Track should ship a small multi-file ruleset under `.cursor/rules/`, not a single monolithic port.
- Keep only Track invariants always-on; move work-loop, task-file, creation, and validation details into narrower requested or auto-attached rules.
- Reuse Track's task/project model and validation commands directly, but adapt Claude-specific skill wrappers instead of copying them verbatim.
- Local Cursor IDE/CLI smoke tests should precede any marketplace or distribution work.
