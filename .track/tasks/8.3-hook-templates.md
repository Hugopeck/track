## Context
Two git hook templates that init deploys to adopting repos:

**commit-msg** — bash-only conventional commit linter.
- Regex: `^(feat|fix|docs|refactor|test|ci|chore)(\(.+\))?!?: .+`
- Pure bash 3.2+, zero external deps
- Clear error message showing expected format on rejection

**post-commit** — event emitter. Hook emits raw events; attribution is server-side.
- Thin bash wrapper: calls `~/.track/bin/track emit post-commit` if binary present (fire-and-forget)
- Fallback (binary absent): appends minimal JSON line to `.track/events/log.jsonl` (gitignored)
- Never blocks the commit; no scope matching in hook — attribution is the server's job
- Payload fields: type, version, timestamp, repo, branch, commit_sha, changed_files, conventional_commit
- Per `.track/specs/event-contract.md` (8.1); task_id/attribution_source are runtime-enriched by server
- Does NOT call `track_match_files_to_task()` — removed; server handles deterministic + LLM attribution

Both hooks live in `skills/init/assets/hooks/`.

## Acceptance Criteria
- [ ] `skills/init/assets/hooks/commit-msg` validates conventional commit format (pure bash 3.2+)
- [ ] `skills/init/assets/hooks/post-commit` delegates to `track emit` binary when available
- [ ] Post-commit fallback appends minimal valid JSON to `.track/events/log.jsonl` when binary absent
- [ ] Post-commit never blocks the commit (fire-and-forget)
- [ ] Event payload matches `.track/specs/event-contract.md` wire format (no task_id / attribution fields — those are runtime-enriched)
- [ ] `tests/test-commit-lint.sh` tests accept/reject cases for commit-msg
- [ ] `tests/test-event-contract.sh` validates JSON output format (both binary path and fallback path)
- [ ] Hook wrapper itself is bash 3.2+ compatible
