## Problem Statement

`skills/work/SKILL.md` had grown into Track's most failure-prone skill. It
combined queue selection, quick operations, lifecycle script calls, PR handoff,
and evidence reporting in one entry-point protocol. Agents frequently skipped
`track-start.sh` or `track-ready.sh`, misclassified `pick` vs `resume`, and
fabricated file+line verification. The skill also forced work into Track too
early, even when exploratory or opportunistic work should have stayed untracked.

## Existing Documented Truth About Untracked Activity

- `TRACK.md` and `AGENTS.md` already say untracked activity is first-class.
- `.track/specs/event-contract.md` already defines `task_id: null` as untracked,
  not lost activity.
- `.track/specs/pr-instructions.md` already includes an untracked PR template.
- PR lint and validation already tolerate PRs that are not Track PRs.

The gap was not runtime support. The gap was that `/track:work` still behaved as
if every session had to begin with queue selection or immediate task attachment.

## New `work` Skill Shape

The redesigned skill is a work-session protocol, not a queue-selection
protocol.

It supports three modes:

- `tracked` — a task is known
- `untracked` — no task should be forced yet
- `empty` — no open tasks exist

It does not support `pick`, standalone `link`, or standalone `context` as
top-level entry modes.

The skill now owns:

1. Reading Track state, branch state, and PR state
2. Running tracked sessions when a task is known
3. Allowing untracked sessions when no task should be forced yet
4. Promoting untracked work to tracked work only through deterministic auto-patch
5. Starting tracked work with `bash .track/scripts/track-start.sh {id}`
6. Opening or continuing draft PRs
7. Persisting notes and plans for tracked work

The skill no longer owns queue selection, quick-operation branching, or direct
`review` writes.

## Deterministic Auto-Patch Hatch

This preserves the useful part of old `pick` mode without reintroducing queue
selection.

The hatch is allowed only when all are true:

1. Exactly one deterministic candidate task exists
2. The candidate task is not terminal or blocked
3. The current work is cohesive with the task's intent
4. Promotion does not create a second independent goal

Deterministic signals are limited to:

- branch or PR linkage that already resolves to one task
- changed files matching exactly one task `files:` scope

If the match is ambiguous or the work would normally be split, the session stays
untracked.

Promotion flow:

1. State the promotion explicitly
2. Run `bash .track/scripts/track-start.sh {id}` if needed
3. Append a task note describing the transition
4. Emit `track.link` if retroactive attribution is required
5. Replace the untracked PR body header with `Track-Task: {id}`

## Automation Boundary

Keep agent-owned:

- `bash .track/scripts/track-start.sh {id}` for entering `active`

Move out of the skill contract:

- `bash .track/scripts/track-ready.sh {id}`

Tracked review handoff is now documented as:

1. verify acceptance criteria with evidence labels
2. mark the PR ready for review
3. let `track-status-sync` write `review`

Merged completion remains automation-owned.

## Docs and Migration Impact

Canonical docs must now say:

- no task defaults to `untracked`, not failure
- `/track:work` no longer selects from `TODO.md`
- tracked work uses `track-start.sh`
- tracked review transitions come from PR lifecycle automation
- untracked PRs are valid until one deterministic task match exists

Runtime compatibility stays intact for now. `track-ready.sh` remains present as a
compatibility wrapper, but the skill no longer teaches it.

## Test Plan

Update docs and skill regression coverage to assert:

- `pick` mode is gone
- `tracked`, `untracked`, and `empty` are documented
- quick operations are gone from the skill
- `track-ready.sh` is not taught in the skill
- `track-start.sh` is still taught
- deterministic auto-patch is documented
- README and setup docs describe untracked work as valid

Retain runtime and status-sync coverage for existing compatibility wrappers.

## Follow-Up Requirements

Track should eventually expose tracked vs untracked activity counts.

That follow-up must decide:

- whether counts are branch-level, day-level, or repo-level
- whether the source of truth is local JSONL only, future server summaries, or both
- where those counts surface in Track views or reporting

This is a product requirement created by the redesign, but it is not in scope
for this skill implementation.
