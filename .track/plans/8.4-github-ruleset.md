# Plan for Task 8.4 — GitHub Ruleset Template

## Summary
Implement task `8.4` by adding a static ruleset payload at `skills/init/assets/track-ruleset.json:1` and a focused shell test at `tests/test-ruleset.sh:1`. Keep this task scoped to the asset + test only; the user-facing “requires repo admin / skip if unavailable” explanation is deferred to task `8.6` when init actually applies the ruleset. The ruleset should target `main` and `master`, enforce PR-based merges, block deletion and force-pushes, require linear history, and require exactly the `track-validate` and `track-pr-lint` checks.

## Scope
- Add `skills/init/assets/track-ruleset.json:1`.
- Add `tests/test-ruleset.sh:1`.
- Do **not** modify `skills/init/assets/install-manifest.json:1`; task `8.4` only creates the asset, and task `8.6` is already the place where init consumes and deploys it.
- Do **not** modify `skills/init/SKILL.md:1`, `README.md:1`, or `TRACK.md:1` in this task; the admin/skip behavior documentation is intentionally deferred to `8.6`.

## Implementation Plan
1. Create `skills/init/assets/track-ruleset.json:1` as a hand-authored, stable JSON file with these exact top-level fields:
   - `name`: `"Track Protection"`
   - `target`: `"branch"`
   - `enforcement`: `"active"`
   - `conditions.ref_name.include`: `["refs/heads/main", "refs/heads/master"]`
   - `conditions.ref_name.exclude`: `[]`
   - `rules`: array of five rule objects in this order:
     1. `{ "type": "deletion" }`
     2. `{ "type": "non_fast_forward" }`
     3. `{ "type": "required_linear_history" }`
     4. `{ "type": "pull_request", "parameters": { ... } }`
     5. `{ "type": "required_status_checks", "parameters": { ... } }`

2. Use these exact `pull_request.parameters` values in the JSON:
   - `dismiss_stale_reviews_on_push`: `true`
   - `require_code_owner_review`: `false`
   - `require_last_push_approval`: `false`
   - `required_approving_review_count`: `0`
   - `required_review_thread_resolution`: `false`
   - Omit `allowed_merge_methods`
   - Omit `required_reviewers`

3. Use these exact `required_status_checks.parameters` values:
   - `strict_required_status_checks_policy`: `true`
   - `required_status_checks`:
     - `{ "context": "track-validate" }`
     - `{ "context": "track-pr-lint" }`
   - Omit `integration_id` for both checks
   - Omit `do_not_enforce_on_create`

4. Explicitly **do not** include `conventional-commit-lint` in the ruleset, even though an older project plan mentioned it. The task file at `.track/tasks/8.4-github-ruleset.md:1` is the current source of truth and only requires `track-validate` and `track-pr-lint`.

5. Keep the JSON deliberately minimal:
   - No `bypass_actors`
   - No `workflows` rule
   - No metadata restrictions
   - No org-only reviewer/team configuration

## Test Plan
Create `tests/test-ruleset.sh:1` in the repo’s existing shell-test style (`set -euo pipefail`, PASS/FAIL counters, human-readable output). Because this uses shell-only validation, the test should be a deterministic structure smoke test rather than a full JSON parse.

Add these assertions:
- File exists and is non-empty.
- First non-whitespace character is `{` and last non-whitespace character is `}`.
- Contains `"name": "Track Protection"`.
- Contains `"target": "branch"` and `"enforcement": "active"`.
- Contains both `refs/heads/main` and `refs/heads/master`.
- Contains each required rule type exactly once:
  - `"type": "deletion"`
  - `"type": "non_fast_forward"`
  - `"type": "required_linear_history"`
  - `"type": "pull_request"`
  - `"type": "required_status_checks"`
- Pull-request block contains:
  - `"dismiss_stale_reviews_on_push": true`
  - `"required_approving_review_count": 0`
  - `"require_code_owner_review": false`
  - `"require_last_push_approval": false`
  - `"required_review_thread_resolution": false`
- Status-check block contains:
  - `"strict_required_status_checks_policy": true`
  - exactly two `"context"` entries
  - `"context": "track-validate"`
  - `"context": "track-pr-lint"`
- Does **not** contain:
  - `conventional-commit-lint`
  - `integration_id`
  - `allowed_merge_methods`
  - `required_reviewers`

Implementation detail for the shell-only test:
- Use `grep -F` for literal assertions.
- Use `grep -c` for exact-count checks.
- Normalize leading/trailing whitespace with `sed`/`tr` only if needed.
- Avoid Bash 4-only features so the test stays compatible with the repo’s Bash 3.2 baseline.

## Validation Steps
Run these after implementation:
- `bash tests/test-ruleset.sh`
- `bash tests/run-all.sh`

## Public Interfaces / Contracts
- **New asset contract:** `skills/init/assets/track-ruleset.json:1` becomes the canonical repository-ruleset payload that task `8.6` will POST via `gh api` later.
- **New verification surface:** `tests/test-ruleset.sh:1` becomes the guardrail that prevents drift in rule names, targeted branches, and required status-check contexts.
- **No current runtime behavior change:** init does not consume this file until `8.6`, so `8.4` is additive and non-breaking.

## Edge Cases and Failure Modes
- Target both `main` and `master`; do not assume only one default branch name.
- Do not rely on org/team reviewer features; those are not portable to user-owned repos and are unrelated to this task’s acceptance bar.
- Keep enforcement `active`, not `evaluate`, because this task is specifically about merge-gate enforcement.
- Treat the admin/permission gate as out of scope for `8.4`; `8.6` should handle it when it actually creates the ruleset.

## Assumptions and Defaults
- **Chosen doc boundary:** user-facing admin/skip documentation is deferred to `8.6`; `8.4` stays asset/test only.
- **Chosen check list:** require only `track-validate` and `track-pr-lint`; older planning notes mentioning `conventional-commit-lint` are superseded by the task file.
- **Chosen test strategy:** shell-only structure checks, not parser-backed JSON validation.
- **Chosen ruleset name:** `"Track Protection"` for a stable, reusable default.
- **Chosen strictness:** `strict_required_status_checks_policy: true`.
- **Chosen omission policy:** leave out optional fields unless they are necessary to satisfy acceptance criteria.
